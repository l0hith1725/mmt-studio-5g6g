# N26 — EPC ↔ 5GC Inter-System Handover

The N26 interface lets a single-registration UE move between an
**MME** (4G) and an **AMF** (5G) without a fresh attach: the source
side derives a **mapped context** (KASME or KAMF, mapped EPS bearers
or PDU sessions, UE info), pushes it across N26, and the target side
consumes it on the next ATTACH / TAU / Service Request. This page
documents the operator-visible projection of N26 in the SA Core build:
an audit log on the AMF side, a mapped-context cache on the MME side,
and a REST surface tying the two together.

# Part A — Functional

## A.1 Why N26?

5G ↔ 4G mobility has two normative variants in TS 23.501 §5.17:

| Mode | Spec | Handover behaviour |
|------|------|--------------------|
| **With N26** | §5.17.2.2 | Mapped context is pushed AMF→MME (5G→4G) or fetched AMF←MME (4G→5G via S1) so the UE skips a full re-authentication. |
| **Without N26** | §5.17.2.3 | UE re-attaches in the target system. Out of scope here — this surface only exists for the N26 path. |

Single-registration UEs (TS 23.501 §5.17.2.2.1) are the in-scope
target. Dual-registration mode is a TODO on the GMM side, unrelated
to this surface.

## A.2 Reference points

| Reference point | Peer | Wire | Spec | Status here |
|-----------------|------|------|------|-------------|
| **N26** | MME ↔ AMF | mapped-context push (no public encoding in TS 23.502) | TS 23.501 §5.17.2.2; TS 23.502 §4.11 | In-process call; both halves live in this build |
| **EPS NAS** | UE → MME | NAS over S1AP / RRC | TS 24.301 | EMM/ESM handlers in `access/epc` |
| **5G NAS** | UE → AMF | 5GMM over NGAP / RRC | TS 24.501 | GMM handler in `nf/amf/gmm` |

The actual NAS / S1 / NGAP signalling — Authentication, SMC, the
Handover Required / Handover Command exchange, the EPS bearer-to-PDU
mapping — lives in `nf/amf/gmm` and the EPC handler. This package's
job is the **audit trail** + **mapped-context cache** + **operator
panel** that lets the GUI / tester drive and observe a handover.

## A.3 Operator-visible behaviours

### A.3.1 AMF audit log

Every N26 handover produces at least one row in `n26_handover_log`,
keyed on `imsi` with `(source_rat, target_rat, status)` matching the
schema CHECK constraints (only `4G`/`5G` and `initiated`/`completed`/
`failed`). The audit row is the first thing written when the handover
decision is taken; the second thing is the status update once the peer
ACKs (or times out).

### A.3.2 MME mapped-context cache

For 5G→4G handover the AMF pushes a mapped EPS context (KASME, EPS
bearers, UE info) into an in-memory cache keyed by IMSI; the EPS attach
handler consumes it on the next matching ATTACH/TAU within the **120 s
TTL** window (Rel-15+ recommendation; see `n26.go:57-60`). Past TTL,
the row is evicted lazily on the next read or by `CleanupExpired`.

For 4G→5G the cache is a *forwarding shim* only — the AMF derives the
5G context server-side from the still-attached S1 leg.

### A.3.3 GUI panel

`/n26.html` (Bootstrap accordion entry) renders three sections:

- Counter cards: 5G UE contexts, 4G UE contexts, N26 enabled/disabled.
- Trigger handover form: `imsi` + direction (`5g-to-4g` | `4g-to-5g`).
- Handover history table from `/api/n26/handovers?limit=20`.

## A.4 Operator REST API (`/api/n26/*`)

| Method | Path | Purpose |
|--------|------|---------|
| GET    | `/api/n26/status` | `{n26_enabled, amf_ue_contexts, mme_ue_contexts, context_ttl_seconds, audit, …}` for the dashboard |
| POST   | `/api/n26/handover/5g-to-4g` | derive mapped context → push to MME → log + complete audit; body `{imsi}` |
| POST   | `/api/n26/handover/4g-to-5g` | forward intent to AMF → log + complete audit; body `{imsi}` |
| GET    | `/api/n26/handovers` | full audit log (most recent first) |
| GET    | `/api/n26/handovers/{imsi}` | per-IMSI audit history |
| GET    | `/api/n26/contexts` | mapped-context cache occupancy + TTL |
| POST   | `/api/n26/contexts/cleanup` | evict every TTL-expired row |
| POST   | `/api/n26/contexts/{imsi}/expire` | evict one cached entry |
| GET    | `/api/n26/stats` | `{total, initiated, completed, failed, 4g_to_5g, 5g_to_4g}` |

### A.4.1 Bodies and shapes

- **Handover** request: `{"imsi": "<imsi>"}`. Unknown IMSIs return
  HTTP 404 (the route hits `crud.UEGetByIMSI` first).
- **Handover** response: `{success, audit_id, imsi, source_rat,
  target_rat, mme_ue_s1ap_id?|amf_ue_ngap_id?}`. The synthetic
  peer-side identifier echoes the audit row id today; live allocation
  comes from the EPC / AMF UE-context manager when the surface is
  promoted to a live handover.
- **Audit** row: `{id, imsi, source_rat, target_rat, source_ue_id,
  target_ue_id, timestamp, status}` — exactly the schema in
  `db/schemas/domains.go::n26_handover_log`.

# Part B — Design

## B.1 Process layout

```
┌──────────── AMF process ────────────┐    ┌──────────── MME shim ────────────┐
│                                     │    │                                  │
│  nf/amf/n26 (audit + stats)         │    │  access/epc/mme/n26 (cache)      │
│   ├── LogHandover(imsi,src,tgt,st)  │    │   ├── ReceiveContextFromAMF(...)  │
│   ├── MarkCompleted(rowID)          │    │   ├── GetMappedContext(imsi)      │
│   ├── MarkFailed(rowID)             │    │   ├── ConsumeMappedContext(imsi)  │
│   ├── List() / ListForIMSI(imsi)    │    │   ├── ForwardContextToAMF(imsi)   │
│   └── Stats() / Status()            │    │   ├── GetN26Status()              │
│         │                           │    │   └── CleanupExpired()            │
│         ▼                           │    │           │                      │
│  n26_handover_log (DB)              │    │  in-memory: mappedContexts[imsi] │
│   id, imsi, source_rat, target_rat, │    │  TTL = n26CTXttl (120s)          │
│   source_ue_id, target_ue_id,       │    │                                  │
│   timestamp, status                 │    │                                  │
│                                     │    │                                  │
└─────────────────────────────────────┘    └──────────────────────────────────┘
                  ▲                                       ▲
                  │                                       │
                  └── webservice/app/routes_n26.go  ──────┘
                                JSON / HTTP via OAM panel
```

The two halves cohabit a single process today: the route handlers
import both `nf/amf/n26` and `access/epc/mme/n26` directly. When the
EPC handler is split into a separate process the cache will move with
it; the AMF audit log stays put.

## B.3 DB schema (`n26_handover_log`)

```sql
CREATE TABLE IF NOT EXISTS n26_handover_log (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  imsi            TEXT NOT NULL,
  source_rat      TEXT NOT NULL CHECK (source_rat IN ('4G', '5G')),
  target_rat      TEXT NOT NULL CHECK (target_rat IN ('4G', '5G')),
  source_ue_id    INTEGER,
  target_ue_id    INTEGER,
  timestamp       TEXT NOT NULL DEFAULT (datetime('now')),
  status          TEXT NOT NULL DEFAULT 'initiated'
                  CHECK (status IN ('initiated', 'completed', 'failed'))
);
CREATE INDEX IF NOT EXISTS idx_n26_ho_imsi ON n26_handover_log(imsi);
CREATE INDEX IF NOT EXISTS idx_n26_ho_ts   ON n26_handover_log(timestamp);
```

The CHECK constraints are the canonical vocabulary — both the Go and
Python sides repeat the same string literals.

## B.4 Handover sequence (operator surface today)

### B.4.1 5G → 4G

```
GUI / tester                  routes_n26.go              nf/amf/n26       access/epc/mme/n26
     │  POST /handover/5g-to-4g    │                          │                  │
     │                              │ crud.UEGetByIMSI         │                  │
     │                              │ — 404 if absent          │                  │
     │                              │ LogHandover(             │                  │
     │                              │   imsi, 5G→4G, initiated)│                  │
     │                              │ ─────────────────────────►                  │
     │                              │     id = NN              │                  │
     │                              │ ReceiveContextFromAMF    │                  │
     │                              │   (imsi, kasme=[],       │                  │
     │                              │    bearers=[], ueInfo)   │                  │
     │                              │ ──────────────────────────────────────────►
     │                              │     stored                                  │
     │                              │ MarkCompleted(id)        │                  │
     │                              │ ─────────────────────────►                  │
     │                              │     row → completed      │                  │
     │ {success, audit_id, imsi,    │                          │                  │
     │  source_rat=5G, target_rat=4G,                         │                  │
     │  mme_ue_s1ap_id}             │                          │                  │
     ◄──────────────────────────────                                              │
```

### B.4.2 4G → 5G

Same shape, except the MME-side call is `ForwardContextToAMF(imsi)`
(stub today) and the response carries `amf_ue_ngap_id` instead of
`mme_ue_s1ap_id`.

### B.4.3 Live signalling (deferred)

The on-wire NAS / S1 / NGAP / mapped-context derivation is not yet
wired here. When promoted, the route's "log → push → mark completed"
sequence will be replaced by:

1. `LogHandover(initiated)`.
2. AMF GMM derives mapped EPS context from the live UE state.
3. `ReceiveContextFromAMF(...)` with real KASME + bearers.
4. EPC handler triggers Handover Required → MME ACK loop.
5. On ACK, `MarkCompleted`; on timeout, `MarkFailed`.

Step 3 is the only hop that lives in this surface today.
